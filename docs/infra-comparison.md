# インフラ構成 比較ドキュメント

対象規模：ユーザー数 最大5名、ボット数 数十程度

---

## AWS 無料枠での運用可否

### 各サービスの無料枠

| サービス | 無料枠 | 5名・常時稼働ボットでの判定 |
|----------|--------|----------------------------|
| **Lambda** | 月100万リクエスト、40万GB秒（永続） | ✗ 常時稼働ボットに不向き。コールドスタートあり、タイムアウト制限あり |
| **DynamoDB** | 25GB、WCU25/RCU25（永続） | △ 低頻度なら無料枠内。高頻度トレードで超過リスクあり |
| **API Gateway** | 月100万リクエスト（永続） | △ ボットが数秒ごとにAPIを叩くと超過。5名・低頻度なら範囲内 |
| **Cognito** | 月5万MAU（永続） | ✓ 5名なら完全無料 |
| **S3 + CloudFront** | 50GB/月、200万リクエスト/月（初年度のみ） | ✓ ダッシュボード程度なら無料枠内 |
| **ECS Fargate** | 無料枠なし | ✗ 常時稼働ボット用途。最小$15〜$20/月 |

### 常時稼働ボットがLambdaに向かない理由

- **タイムアウト上限**：最大15分。ボットは24時間稼働が必要
- **コールドスタート**：起動遅延が発生し、価格変動への即応が難しい
- **WebSocket維持不可**：取引所のリアルタイムデータ受信に支障
- **代替案**：ボット実行層だけ ECS Fargate Spot（約7割引）または Fly.io に分離

### AWS構成で無料に近づける場合の設計

```
[フロントエンド]  S3 + CloudFront      → ほぼ無料
[認証]           Cognito               → 無料（5名）
[API]            Lambda + API Gateway  → 低頻度なら無料
[DB]             DynamoDB              → 低頻度なら無料
[ボット実行]     ECS Fargate Spot      → $5〜$10/月（ここだけ有料）
```

**月額目安：$5〜$15/月**（ボット実行層のみ課金）

ただし構成が複雑になる点に注意（IAM設定、VPC、CloudFormationなど学習コストあり）。

## IPv6問題について

2024年以降、AWSがIPv4アドレスを有料化したことで多くのサービスが影響を受けている。
常時稼働ボット（Python）はDB接続が必須なため、接続経路のIPv6対応状況が重要。

| サービス | IPv6対応 | 備考 |
|----------|----------|------|
| **Supabase** | △ | 直接接続はIPv6のみ。IPv4接続は +$4/月 or 無料のコネクションプーラー経由 |
| **Railway** | ○ | 内部接続はIPv6、外部はIPv4対応。同一Railway内なら問題なし |
| **Fly.io** | △ | 内部はIPv6専用。外部IPv4は有料。Prismaなど一部ライブラリで問題報告あり |
| **Render** | △ | IPv6サポート明記なし。Supabaseへの接続で問題報告あり |
| **AWS RDS** | ○ | IPv4/IPv6デュアルスタック対応。ただしIPv4は課金対象 |
| **GCP Cloud SQL** | ○ | IPv4/IPv6対応。フルマネージド |
| **Neon** | ○ | IPv4/IPv6両対応 |

## Redis（キャッシュ・状態管理）の要否

5名・数十ボット規模では **Redis不要** と判断。

| 用途 | Redisなしの代替 |
|------|----------------|
| ボット状態管理 | PostgreSQLまたはDynamoDBのステータスカラムで管理 |
| リアルタイム通知 | WebSocket + DB polling（数秒間隔）で十分 |
| セッション管理 | JWTトークンで代替 |

## コスト削減の設計方針

1. **フロントエンドとバックエンドを分離**
   - フロントはVercel/S3（無料枠）、バックエンドのみ有料サーバーへ

2. **DB・バックエンドを同一プラットフォームに集約**
   - 内部接続にすることでIPv6問題を回避し、外部通信コストも削減

3. **バックテストはオンデマンド実行**
   - 常時起動不要。リクエスト時のみ実行してリソース節約

4. **取引所データはAPIキャッシュ**
   - 同じデータの重複取得を避け、APIレート制限とコストを削減

5. **Redisは当面不使用**
   - PostgreSQLで代替し、必要になった時点で追加
