# インフラ構成 比較ドキュメント

対象規模：ユーザー数 最大5名、ボット数 数十程度

---

## IPv6問題について

2024年以降、AWSがIPv4アドレスを有料化したことで多くのサービスが影響を受けている。
常時稼働ボット（Python）はDB接続が必須なため、接続経路のIPv6対応状況が重要。

| サービス | IPv6対応 | 備考 |
|----------|----------|------|
| **Supabase** | △ | 直接接続はIPv6のみ。IPv4接続は +$4/月 or 無料のコネクションプーラー経由 |
| **Railway** | ○ | 内部接続はIPv6、外部はIPv4対応。同一Railway内なら問題なし |
| **Fly.io** | △ | 内部はIPv6専用。外部IPv4は有料。Prismaなど一部ライブラリで問題報告あり |
| **Render** | △ | IPv6サポート明記なし。Supabaseへの接続で問題報告あり |
| **AWS RDS** | ○ | IPv4/IPv6デュアルスタック対応。ただしIPv4は課金対象 |
| **GCP Cloud SQL** | ○ | IPv4/IPv6対応。フルマネージド |
| **Neon** | ○ | IPv4/IPv6両対応 |

**結論：** SupabaseのIPv4アドオン（$4/月）は許容範囲だが、Railway内で完結させる方がシンプル。

## Redis（キャッシュ・状態管理）の要否

5名・数十ボット規模では **Redis不要** と判断。

| 用途 | Redisなしの代替 |
|------|----------------|
| ボット状態管理 | PostgreSQLのステータスカラムで管理 |
| リアルタイム通知 | WebSocket + DB polling（数秒間隔）で十分 |
| セッション管理 | JWTトークンで代替 |

将来的にユーザー・ボット数が増えた場合に追加を検討。

## コスト削減の設計方針

1. **フロントエンドとバックエンドを分離**
   - フロントはVercel（無料枠）、バックエンドのみ有料サーバーへ

2. **DB・バックエンドを同一プラットフォームに集約**
   - 内部接続にすることでIPv6問題を回避し、外部通信コストも削減

3. **バックテストはオンデマンド実行**
   - 常時起動不要。リクエスト時のみ実行してリソース節約

4. **取引所データはAPIキャッシュ**
   - 同じデータの重複取得を避け、APIレート制限とコストを削減

5. **ログ・監視は無料枠で**
   - Grafana Cloud（無料枠）または Sentry（無料枠）を活用

6. **Redisは当面不使用**
   - PostgreSQLで代替し、必要になった時点で追加
